# 苏适酒店 agent 数字员工 — API 接入手册

> 本手册回答一个问题：**每个 agent 要真正跑在两店 124 间房上，需要接入哪些外部 API？**
>
> 按 agent 拆解 → 每项列候选厂商 / 文档入口 / 申请周期 / 工程量 / 合规提示。

---

## 1. 三阶段落地路径

| 阶段 | 周期 | 范围 | 目标 |
|------|------|------|------|
| **MVP** | 1-2 月 | 数字前台 · 在线客服 · OTA 运营师 | 把订单、客户、渠道打通，立刻见收益 |
| **可运营** | 3-4 月 | 客房管家 · 收益管理师 · 工程运维 | 体验和能耗降本，IoT 开始进场 |
| **深度智能** | 5-6 月 | 宾客画像 · 数字店长 · 数字营销 · 财务风控 · 供应链 · 数字质检 | 全局优化、跨店对标、合规稽核 |

> **强依赖关系**：PMS/CRS 必须第一步解决；OTA 三大平台申请周期长，要 D-Day 前 6 周启动；支付商户号 2 周；企微应用当天能搞定。

### 3.4 PMS 供应商：深圳捷信达（JXD）

**业主已确定选型**：深圳市捷信达信息技术有限公司。本项目不考虑其他 PMS。

#### 基本信息

- **全名**：深圳市捷信达信息技术有限公司（业内简称"**捷信达**"/ JXD）
- **产品线**：酒店 PMS，国内老牌厂商，中高端酒店覆盖度高
- **部署形态**：以本地部署为主，部分新客户可用云版（需跟对方确认本次是哪一种）
- **接入方式**：通常 REST API / Web Service（签约后看官方文档）
- **认证**：`AppKey + AppSecret` 签名（HMAC-SHA256 或 MD5，版本差异）
- **Webhook**：支持订单 / 房态变更推送（需单独开通）

#### 必须从捷信达拿到的关键资料（业主 or 项目方向对方索要）

- [ ] **API 完整文档**（接口清单 + 请求 / 响应示例）
- [ ] **数据字典**（订单 / 房态 / 客人 / 账单的所有字段定义）
- [ ] **Webhook 事件列表** + 订阅方式 + 签名校验说明
- [ ] **沙箱环境地址** + 测试账号（包括测试酒店的 PropertyCode）
- [ ] **AppKey / AppSecret**（生产 + 测试各一套）
- [ ] **IP 白名单登记方式**（我方服务器公网出口 IP）
- [ ] **限流规则**（单账号 QPS 上限）
- [ ] **错误码表**
- [ ] **PMS 版本号**（不同版本 API 有差异）
- [ ] **技术对接工程师**（姓名 / 电话 / 微信 / 邮箱）
- [ ] **故障 SLA + 应急联系方式**

#### 签约前必须跟捷信达确认的 8 个关键问题

| 问题 | 为什么重要 |
|------|-----------|
| API 形式是 REST 还是 SOAP / XML-RPC？ | 直接影响工程量和语言选型 |
| Webhook 支持哪些事件？失败重试策略？签名算法？ | 决定数据实时性和幂等设计 |
| 房价字段是**含税**还是**不含税**？多币种怎么处理？ | 对账必须一致，否则财务炸 |
| 是否支持批量查询？单次最大条数？ | 影响夜间对账和数据仓 ETL 性能 |
| 历史数据能全量导出做本地仓吗？ | 影响收益 Agent 的训练数据基础 |
| 敏感字段（身份证 / 电话）怎么授权？是否默认脱敏？ | 合规底线 |
| 能否订阅数据库 CDC（如 Binlog）？ | 高频场景的兜底方案 |
| 故障 SLA + 技术支持响应时间？7×24 还是工作日？ | 生产运维关键 |

#### 本项目接入路径

无论捷信达是本地部署还是云版，架构上都走 **§3.7 的反腐层**：

- 我方搭一个**统一 PMS 网关服务**，内部实现一个 `JXDAdapter`
- 12 个 Agent 只跟网关的统一领域模型（Reservation / Guest / Room / Folio）打交道，不直连捷信达
- 这样做的目的：未来扩店 / 换 PMS / 升级版本，不动 Agent 层；同时可以做新老 PMS 双写灰度


### 3.5 上线前业主准备清单

跟 PMS 供应商签约到真正对接之间，业主方要准备：

- [ ] **商务合同**：PMS 采购 / SaaS 订阅合同
- [ ] **酒店唯一编码**：每家店的 `HotelId / PropertyCode`（PMS 分配）
- [ ] **技术对接授权**：书面授权我方作为技术对接方
- [ ] **房型编码映射表**：PMS 房型代码 ↔ 本项目 agent 代码（BKT / CKR / DKL 等）
- [ ] **会员等级定义**：PMS 内会员体系 level 代码 + 权益
- [ ] **渠道挂牌映射**：每个 OTA 平台上架的房型对应 PMS 房型
- [ ] **网络白名单**：我方服务器公网 IP 报 PMS 白名单（本地部署型必需）
- [ ] **测试环境授权**：开通沙箱或测试酒店
- [ ] **签名密钥 / OAuth 凭证**：AppKey / AppSecret 或 OAuth Client ID / Secret
- [ ] **数据迁移方案**（如换 PMS）：历史订单 / 客人档案 / 积分如何迁移

### 3.6 常见接入坑点

| 坑 | 规避方式 |
|----|---------|
| **时区** | 统一 UTC 存储，展示转 Asia/Shanghai |
| **状态机差异** | 统一状态枚举，每家 PMS 单独写映射 |
| **字段命名** | 抽象**领域模型**（反腐层），Agent 不直连 PMS 模型 |
| **价格含税** | 合同里**写死**口径 + 文档标注 |
| **Webhook 不可靠** | 实现**幂等接收** + **兜底轮询**（5 分钟 diff 一次） |
| **大批量限流** | PMS 通常 5-10 QPS，**夜间全量 + 日间增量** |
| **同名异义** | `checkinDate` 在不同 PMS 可能是预抵日 vs 实际入住日，要问清 |
| **敏感字段授权** | 证件号 / 银行卡要单独"敏感字段读权限"，默认脱敏 |

### 3.7 推荐架构：反腐层（Anti-Corruption Layer）

```
┌──────────────┐     ┌─────────────────┐
│  12 个 Agent │ ──→ │  统一 PMS 网关   │ ──→ PMS 供应商
└──────────────┘     └─────────────────┘        (别样红 / 住哲 / 金天鹅)
                             ↓
                  统一领域模型
                  (Reservation / Guest / Room / Folio)
                             ↓
                  本地数据仓
                  (PostgreSQL + ClickHouse)
```

**好处**：
- 换 PMS 只动一层
- Agent 只认统一领域模型
- 可做**双写 / 灰度**（新老 PMS 并行跑）

**关键组件**：
- `PMSAdapter` 接口：每家 PMS 一个实现
- `EventNormalizer`：Webhook 格式归一
- `ReconcileJob`：定时全量对账（防 Webhook 丢失）

### 3.8 下一步具体动作

业主已确认走**深圳捷信达**，下一步路径：

5. **Week 4**：两店全量接入，其他 11 个 Agent 陆续挂入统一领域模型

## 6. 合规与风险清单

### 数据合规
- **PII 加密**：身份证号、手机号、人脸图像必须加密存储（SM4 或 AES-256）
- **公安联网核查**：入住信息 30 分钟内上报公安治安平台（接入省级公安旅业系统）
- **等保 2.0**：全系统按三级等保建设
- **数据出境**：不允许 PII 流向境外云/模型

### 业务合规
- ⛔ **不爬 OTA C 端**（携程/美团 App）— 违反 TOS，IP 封禁风险
- ✍️ **点评 AI 回复人工确认** — 防止话术不当
- 💰 **支付预授权必须二次人工审核** — 防钓鱼/错扣
- 🔒 **门禁开锁禁止给客服类 Agent** — 权限最小化

### 接入质量
- 所有外部 API 调用必须有**熔断 + 重试 + 超时**（建议用 Sentinel）
- OTA 接口速率限制严格（1-5 QPS），必须**令牌桶**
- 重要业务（支付、开门）走**同步 + 幂等**，不走 MQ 异步
